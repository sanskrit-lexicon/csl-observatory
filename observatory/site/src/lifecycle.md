# Issue lifecycle & responsiveness

How long issues live in the sanskrit-lexicon org: cohort survival, backlog age, closing latency, and per-repository responsiveness. Object of analysis is the issue/PR record itself — not dictionary content. Source: [`reports/issue_lifecycle.md`](https://github.com/sanskrit-lexicon/csl-observatory/blob/main/reports/issue_lifecycle.md), generated by [`scripts/issue_lifecycle.py`](https://github.com/sanskrit-lexicon/csl-observatory/blob/main/scripts/issue_lifecycle.py).

Time-to-first-response is not computable from the committed snapshot (it carries comment counts, not comment timestamps) — it is an API-gated extension tracked in the roadmap (Workstream G1).

```js
const survival = await FileAttachment("data/issue_lifecycle_survival.csv").csv({typed: true});
const backlog = await FileAttachment("data/issue_lifecycle_backlog.csv").csv({typed: true});
const close = await FileAttachment("data/issue_lifecycle_close.csv").csv({typed: true});
const repos = await FileAttachment("data/issue_lifecycle_repo.csv").csv({typed: true});
```

```js
const openTotal = d3.sum(backlog, d => d.count);
const silentTotal = d3.sum(backlog, d => d.silent);
const old2y = d3.sum(backlog.filter(d => ["2–4 years", "4+ years"].includes(d.bucket)), d => d.count);
const closeIssues = close.filter(d => d.kind === "issue");
const latestClose = closeIssues[closeIssues.length - 1];
const surv365 = survival.filter(d => d.horizon_days === 365);
const pooled365 = d3.sum(surv365, d => d.still_open) / d3.sum(surv365, d => d.observable);
```

## Cohort survival

> **How to read:** Each row is the set of issues opened in one year; each column a horizon after opening. The cell is the share of that cohort still open at that horizon — darker means more of the cohort survived unresolved. Cells are right-censored: a cohort only reports horizons its issues have had time to reach. **Example 1:** A dark 2014 row means the founding-era backlog never got worked off. **Example 2:** If recent rows go pale by the 90-day column, current triage is resolving most of what arrives within a quarter.

```js
const horizonLabel = new Map([[30, "30 d"], [90, "90 d"], [180, "180 d"], [365, "1 y"], [730, "2 y"], [1460, "4 y"]]);
const survCells = survival.map(d => ({...d, horizon: horizonLabel.get(d.horizon_days)}));
```

```js
Plot.plot({
  width,
  height: 420,
  marginLeft: 60,
  x: {label: "Time since opening", domain: [...horizonLabel.values()]},
  y: {label: "Opening cohort", type: "band", tickFormat: d => String(d)},
  color: {type: "linear", scheme: "blues", domain: [0, 100], label: "% still open", legend: true},
  marks: [
    Plot.cell(survCells, {x: "horizon", y: "cohort", fill: "pct_open", inset: 1, tip: true,
      title: d => `Opened ${d.cohort}: ${d.pct_open}% of ${d.observable} still open after ${d.horizon}`}),
    Plot.text(survCells, {x: "horizon", y: "cohort", text: d => `${Math.round(d.pct_open)}%`,
      fill: d => d.pct_open > 55 ? "white" : "black"})
  ]
})
```

> **Conclusion:** Survival is era-dependent, not uniform. The 2014 founding cohort still has roughly half of its issues open a decade later, while mid-era cohorts stabilise near a third. Whatever is not closed in the first weeks tends to stay open for years — the survival curve flattens almost immediately after the 90-day mark.

## Backlog age pyramid

> **How to read:** Currently-open issues bucketed by age at the snapshot. Each bar splits into issues that received at least one comment (blue) and the silent backlog that never got a single reply (amber). **Example 1:** A long "4+ years" bar means the open backlog is dominated by legacy items, not fresh work. **Example 2:** A large amber share in young buckets means new scholars' reports are currently going unanswered — the worst signal for contributor conversion.

```js
const backlogLong = backlog.flatMap(d => [
  {bucket: d.bucket, status: "Commented", count: d.count - d.silent},
  {bucket: d.bucket, status: "Silent (0 comments)", count: d.silent}
]);
```

```js
Plot.plot({
  width,
  height: 280,
  marginLeft: 110,
  x: {label: "Open issues", grid: true},
  y: {label: null, domain: backlog.map(d => d.bucket)},
  color: {domain: ["Commented", "Silent (0 comments)"], range: ["#0969da", "#bf8700"], legend: true},
  marks: [
    Plot.barX(backlogLong, {x: "count", y: "bucket", fill: "status", inset: 1, tip: true}),
    Plot.text(backlog, {x: "count", y: "bucket", text: d => String(d.count), dx: 14}),
    Plot.ruleX([0])
  ]
})
```

> **Conclusion:** The open backlog is old: two thirds of it predates 2022. The actionable slice is the silent backlog — ${silentTotal} issues nobody ever answered. Triaging those (close, label, or reply) is the cheapest single move to make the org look alive to an arriving scholar.

## Time-to-close by closing year

> **How to read:** For issues closed in each year, the line is the median days from opening to closing and the band spans the 25th–75th percentile. **Example 1:** A low, flat median with a wide band means most closures are fast but each year also digests some very old items. **Example 2:** A spike in the band without a spike in the median is a backlog-clearing campaign — old issues being swept, everyday responsiveness unchanged.

```js
Plot.plot({
  width,
  height: 300,
  x: {label: "Closing year", tickFormat: d => String(d)},
  y: {label: "Days to close", grid: true, type: "sqrt"},
  marks: [
    Plot.areaY(closeIssues, {x: "year", y1: "p25", y2: "p75", fill: "#0969da", fillOpacity: 0.15}),
    Plot.lineY(closeIssues, {x: "year", y: "median_days", stroke: "#0969da", strokeWidth: 2, marker: "circle", tip: true,
      title: d => `${d.year}: median ${d.median_days} d (p25 ${d.p25}, p75 ${d.p75}, p90 ${d.p90}) over ${d.n_closed} closed`}),
    Plot.ruleY([0])
  ]
})
```

> **Conclusion:** Everyday responsiveness is genuinely fast — the median closure takes days, not months, in most years. The long tail (p90 around a year) is where the survival problem lives: an issue either gets handled in its first days or joins the multi-year backlog.

## Per-repository responsiveness

> **How to read:** The fifteen repositories with the most issues, sorted by median days-to-close (computed only where at least 20 issues have closed). The dot is the median; the amber count is that repo's silent open issues. **Example 1:** A repo with a low median but many silent opens closes what it engages with, yet lets the rest sit unacknowledged. **Example 2:** A high median on a dictionary repo means a scholar filing a correction there waits longest for resolution.

```js
const repoTop = repos.filter(d => d.median_days_to_close !== null && d.median_days_to_close !== "")
  .sort((a, b) => b.issues - a.issues).slice(0, 15)
  .sort((a, b) => b.median_days_to_close - a.median_days_to_close);
```

```js
Plot.plot({
  width,
  height: 360,
  marginLeft: 130,
  x: {label: "Median days to close", grid: true, type: "sqrt"},
  y: {label: null, domain: repoTop.map(d => d.repo)},
  marks: [
    Plot.ruleY(repoTop, {y: "repo", x: "median_days_to_close", stroke: "#d0d7de"}),
    Plot.dot(repoTop, {x: "median_days_to_close", y: "repo", r: 5, fill: "#0969da", tip: true,
      title: d => `${d.repo}: median ${d.median_days_to_close} d over ${d.closed} closed · ${d.open} open (median age ${d.open_median_age_days} d) · ${d.silent_open} silent`}),
    Plot.text(repoTop, {x: "median_days_to_close", y: "repo", text: d => d.silent_open ? `${d.silent_open} silent` : "", dx: 10, dy: -8, fill: "#bf8700"})
  ]
})
```

> **Conclusion:** Responsiveness differs by an order of magnitude between repositories under the same three maintainers — the difference is attention routing, not capacity. The slow large repos are where an arriving contributor's first issue is most likely to die unanswered, which makes them the priority for the silent-backlog triage above.
